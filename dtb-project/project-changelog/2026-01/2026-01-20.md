## Session 2 - 16:40

### Analysiert
- Langfuse-Verbindung getestet und SSL-Problem behoben
- Trace-Analyse durchgeführt (560 Traces im System)
- dtb-assistant-chat Traces im Detail untersucht
- Evaluierungs-Scores analysiert (Answer Relevance, CRITERIA4-NUMBERS-WITH-IDENTIFIED-SOURCE)
- LLM-as-a-Judge Konfiguration in Langfuse besprochen

### Erkenntnisse aus Trace-Analyse
- **System**: dtb-assistant-chat mit Qwen3-30B-Instruct (vLLM), LangGraph Framework
- **Performance**: Durchschnittliche Latenz 16.4s, Maximum 87.8s
- **Qualität**: Answer Relevance 94%, Numbers-with-Source 79%
- **Token-Verbrauch**: ~95k Input / ~1.7k Output pro komplexe Anfrage (11 LLM-Calls)
- **Benutzer**: 4 aktive User, hauptsächlich DOMINIKMARTIN.WYSS@USB.CH

### Langfuse-Konzepte erklärt
- **Observation Levels**: DEFAULT, ERROR, WARNING, DEBUG (Log-Level pro Observation)
- **Tokens-Spalte**: Input/Output Token-Verhältnis pro LLM-Call
- **Usage-Spalte**: Token-Verbrauch und optional Kosten
- **Score-Spalten (#)**: Aggregierte Evaluierungs-Werte pro Trace
- **Metadata**: SDK-Info, Service-Info, Custom-Daten vom Backend
- **Tags**: Werden vom Backend gesetzt (mis-default, vllm)

### LLM-as-a-Judge Setup
- **EVAL-Scores**: Answer Relevance, CRITERIA4 (von Langfuse LLM-Evaluator)
- **API-Scores**: eval_*, fact_check, response_latency (vom Backend programmatisch)
- Modell-Empfehlung für Evaluator: phi4-reasoning (14B) oder mistral-small (22B)

### Problem: Ollama/vLLM Verbindung mit Langfuse
- nginx Basic Auth vor vLLM blockiert Langfuse-Verbindung
- Langfuse sendet Bearer Token, nginx erwartet Basic Auth
- curl funktioniert mit `-u ollama_user:password`, Langfuse nicht
- Workaround: Google-Verbindung als Test angelegt

### Konfiguration
- `LANGFUSE_VERIFY_SSL=false` - SSL-Verifizierung deaktiviert
- `LOG_LEVEL=DEBUG` - Request-Logging aktiviert

### Dateien
- `.env` - SSL-Verify und Log-Level angepasst

### Kontext
Erste produktive Nutzung des MCP-Servers zur Analyse von Langfuse-Traces.
